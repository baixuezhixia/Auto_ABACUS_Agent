# Agent Rebuild I/O Spec (Simple)

This guide defines the current minimum input and the standard output locations.

## 1) Minimal Input You Need To Prepare

### 1.1 Config file

Use `config.yaml` (copy from `config.example.yaml`) and fill:

- `abacus.executable`
- `abacus.run_mode`
- `abacus.np`
- `abacus.pseudo_dir`
- `abacus.orb_dir`

### 1.2 Materials Project access

Export `MP_API_KEY` in the environment before running.

### 1.3 Structure input

`--structure` is required.

It is not a local file path anymore. It must be one of:

- a Materials Project material ID, such as `mp-149`
- a formula / search term, such as `Si` or `SiO2`

Current selection behavior:

- query Materials Project
- use the first returned match
- record other leading matches in `report.json`
- download the conventional CIF file

## 2) Standard Runtime Output Generated By Agent

Given `--work-dir ./runs_mp`, the outputs are currently:

- `./runs_mp/materials_project/<material_id>.cif`
- `./runs_mp/report.json`

`report.json` is the main machine-readable result.

Key fields:

- `structure.material_id`: selected Materials Project ID
- `structure.cif_path`: downloaded CIF path
- `structure.lattice_type`: currently always `conventional`
- `structure.candidates[]`: leading search candidates
- `tasks[]`: decoded task plan
- `notices[]`: why the workflow stopped before execution

## 3) Current Stop Point

The current pipeline stops after CIF download.

Not implemented yet:

- `CIF -> STRU` conversion
- generation of final ABACUS `INPUT / STRU / KPT`
- ABACUS execution

## 4) Where To Check Results Quickly

After a run, check in this order:

1. `report.json`
2. `materials_project/<material_id>.cif`
3. console notices for candidate selection and stop reason
